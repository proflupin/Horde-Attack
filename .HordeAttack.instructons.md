# Horde Attack Project Guidelines

Horde Attack is a text-based roguelike written in Python.

The code should remain beginner-friendly and educational.

## Before Editing

Always:

1. Explain the requested change.
2. Describe your implementation plan.
3. Wait for my approval before editing files.

## Design Philosophy

Favor:

- simple code
- readable code
- maintainable code

Avoid:

- clever one-liners
- unnecessary abstractions
- overengineering

This project is meant to be understandable by a student.

## Existing Style

Follow the existing coding style.

Use:

- dictionaries for entities
- standalone functions
- descriptive variable names
- clear comments where useful

Do not completely rewrite working systems.

## Adding Features

When adding mechanics:

- integrate with the existing architecture
- minimize unrelated changes
- preserve save compatibility when possible

If a feature requires major restructuring, explain why before doing it.

## Balancing

When suggesting gameplay changes:

Prefer incremental adjustments.

Avoid drastic balance swings.

Explain why a change improves gameplay.

## Debugging

Find the root cause before changing code.

Never make speculative fixes.

If uncertain, ask questions.

## Refactoring

Only refactor code when it clearly improves:

- readability
- maintainability
- performance

Never refactor unrelated code while implementing another feature.

## Documentation

When adding complicated logic, briefly explain:

- what it does
- why it exists

Future me should be able to understand the code months later.

## Communication

If multiple implementations are possible:

- recommend the simplest one first
- explain tradeoffs
- let me decide

Do not silently make architectural decisions for me.
